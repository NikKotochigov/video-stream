## Слайд 2 — Видео ≠ `<video src>`

### На стриме

> В HTML видео — это `<video src="file.mp4">`. В приложении с камерой — **другой путь**. `<video>` в конце, не в начале.

### Под капотом — файл vs живой поток

**Файл (mp4):**
1. Браузер **скачивает** байты по URL.
2. **Демuxer** делит на аудио/видео дорожки.
3. **Decoder** (часто аппаратный) → кадры в память.
4. `<video>` **рисует** кадры. Можно **seek** — все кадры уже известны.

**Живой поток (камера):**
1. Нет URL файла. Кадры **рождаются сейчас** на устройстве.
2. `getUserMedia` → **MediaStream** — «труба» из браузера в JS.
3. Кадры идут **пока не stop()**. Seek невозможен.
4. `<video srcObject=stream>` — **монитор** потока, не файловый плеер.

### Под капотом — почему нельзя `src="camera"`

Атрибут `src` ждёт **адрес ресурса** (URL). У камеры нет URL — есть **устройство захвата**, доступ через браузер + ОС + permissions. Поэтому отдельный API и свойство **`srcObject`**.

### Под капотом — аналогия

| | Файл | Камера |
|---|------|--------|
| | Книга на полке | Прямой эфир |
| Перемотка | Да | Нет |
| Конец | Есть | Только после stop |

### На слайде

Провести по схеме: верхняя строка короткая, нижняя — весь эфир.

### Зритель должен унести

> `<video src>` — для **файла**. Камера — **MediaStream** + **srcObject**.

## Слайд 3 — Фокус эфира

### На стриме

> Что **разберём** и что **сознательно не трогаем** — чтобы не ждать blur в первые 20 минут.

### Под капотом — что входит

| Тема | Суть |
|------|------|
| getUserMedia | Доступ к камере, constraints |
| MediaStream / tracks | Что внутри потока, enable vs stop |
| Canvas + captureStream | Обработка кадров «по-старому» |
| Insertable Streams | Обработка VideoFrame «по-новому» |
| WebRTC loopback | Поток через encoder/decoder в одной вкладке |

### Под капотом — чего нет и почему

- **Blur / virtual background** — отдельные алгоритмы + perf; нужна база по кадрам.
- **ML-сегментация** — TensorFlow Lite, workers; не про getUserMedia.
- **Prod WebRTC** (signaling, TURN) — другой масштаб; loopback учит offer/answer/ICE без сервера.

### Если спросят про blur

## Слайд 4 — План + demo

### На стриме

> Пять шагов в demo слева — один URL. Сейчас покажу вкладку и начнём с камеры.

### Под капотом — что на каждом шаге (кратко)

**Шаг 1 — getUserMedia**  
`navigator.mediaDevices` → permissions → MediaStream → `<video srcObject>`.  
Разберём constraints, `<video>` attrs (autoPlay, playsInline, muted).

**Шаг 2 — tracks**  
Один stream — два preview. `track.enabled` vs `track.stop()`.

**Шаг 3 — Canvas**  
`drawImage` + `canvas.captureStream()` → **новый** MediaStream.

**Шаг 4 — Insertable Streams**  
VideoFrame pipeline; fallback если API нет.

**Шаг 5 — WebRTC**  
Два RTCPeerConnection, local/remote video, ICE connected.

### Действия

1. `localhost:5173`  
2. Меню шагов 1–5  
3. Остановиться на **Шаг 1**

### Под капотом — почему одно приложение

Один **StreamContext** несёт `MediaStream` через все шаги — как в реальном app: камера один раз, дальше обработка и отправка.
